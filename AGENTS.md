# AGENTS.md

## Project Overview

**Profile** is a personal portfolio website built with **Astro 7** and deployed on **Cloudflare Pages**. The site showcases a senior‑level backend engineer's professional brand, highlighting technical expertise, open‑source contributions, and cloud‑native deployment practices.

## Tech Stack

- **Framework:** Astro 7 (static site generation with component‑driven architecture)
- **Runtime:** Cloudflare Pages (edge‑hosted, server‑less functions via Cloudflare Workers)
- **Styling:** Tailwind CSS (utility‑first, JIT compiled)
- **Type‑checking:** TypeScript 5
- **Testing:** Vitest, Playwright (E2E)
- **CI/CD:** GitHub Actions (automated lint, type‑check, test, build, and deploy)

## Architecture

```
+-------------------+      +-------------------+      +-------------------+
|  Astro Sources    | ---> |  Astro Build      | ---> |  Cloudflare Pages |
| (src/, public/)   |      | (astro build)     |      | (static assets)   |
+-------------------+      +-------------------+      +-------------------+
``` 

- **Source Layer** – All UI components, pages, and utilities live under `src/`. Static assets (images, fonts) are stored in `public/`.
- **Build Layer** – `astro dev` runs a local development server; `astro build` compiles the site to a static output in `dist/`.
- **Deploy Layer** – `wrangler pages deploy` publishes the built assets to Cloudflare’s edge network, offering instant global caching and TLS termination.

## Backend Highlights (Senior‑Level Focus)

Although the site is static, it leverages Cloudflare Workers for dynamic capabilities:

- **Server‑less API endpoints** (e.g., contact form handling, GitHub profile fetch) defined in `src/api/` and exposed via `wrangler pages functions`.
- **Edge caching** with custom `Cache-Control` headers to minimize latency and reduce origin load.
- **Secure environment variables** managed through Cloudflare Workers Secrets, keeping API keys out of the repo.
- **Type‑safe request/response contracts** using TypeScript interfaces, ensuring compile‑time correctness.

## Semantic Commit Guidelines

All commits should follow the **Conventional Commits** specification to keep the history clear and machine‑friendly.

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

| Type | When to use |
|------|-------------|
| `feat` | New feature (e.g., adding a new page or API route) |
| `fix` | Bug fix (e.g., correcting a broken component) |
| `docs` | Documentation changes (e.g., updating this AGENTS.md) |
| `style` | Code style changes (formatting, lint fixes) |
| `refactor` | Code restructuring without functional change |
| `perf` | Performance improvements (e.g., cache headers) |
| `test` | Adding or fixing tests |
| `chore` | Build scripts, CI config, dependency upgrades |

**Scope** should be the top‑level directory or logical module (`src`, `public`, `config`, `ci`).

*Example:* `feat(layout): add glassmorphism hero section`

## Release Process
